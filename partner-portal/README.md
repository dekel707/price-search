# הזמנות איתן — פיילוט מבודד

זהו פורטל נפרד לחלוטין מהמערכת הראשית. הוא מיועד לפיילוט שבו איתן מחפש במפרטי מוצרים, בוחר לקוח קיים במסד החדש, רואה שריונים וגיול, ושולח הזמנה לאישור בעלים.

## גבולות ההפרדה

- אין שימוש ב־`DATABASE_URL`, `BLOB_READ_WRITE_TOKEN`, `/api/state` או ב־Vercel Blob של המערכת הראשית.
- כל טבלת נתונים בפורטל מתחילה ב־`partner_` ונמצאת במסד PostgreSQL חדש שהוגדר רק עבור הפרויקט הזה.
- הקטלוג נקרא ב־GET בלבד מ־`TEAM_PORTAL_CATALOG_URL`, ומנוקה לפני החזרה כך שאינו כולל מלאי, מחיר, הזמנות, לקוחות או שריונים של האתר הראשי.
- הזמנות בפיילוט אינן נכנסות אוטומטית לאתר הראשי ולא משנות את הדאשבורד הראשי. זהו גבול בטיחות מכוון.

## הגדרה לפריסה

1. יוצרים ב־Vercel **פרויקט חדש** עם Root Directory `partner-portal`.
2. יוצרים מסד PostgreSQL חדש לפרויקט החדש בלבד (Neon או Supabase) ולא מחברים שום אינטגרציית Storage קיימת.
3. מגדירים את כל המשתנים שב־`.env.example` רק בפרויקט החדש.
4. מגדירים את ה־Cron היומי ב־`vercel.json`; Vercel משתמש ב־`CRON_SECRET` כדי לחתום על הבקשה, והבקשה יוצרת צילום מצב במסד החדש.
5. הבעלים נכנס עם `TEAM_PORTAL_OWNER_PIN`; איתן נכנס עם `TEAM_PORTAL_EITAN_PIN`.

## הגנות נתונים

- לפני ואחרי כל פעולה משנה נתונים נוצר צילום מצב מלא בטבלת `partner_backups`.
- שריון ננעל באמצעות `SELECT ... FOR UPDATE` בתוך טרנזקציה. לכן שתי הזמנות במקביל אינן יכולות למשוך יותר מהכמות השמורה.
- כל משיכה משריון נרשמת ב־`partner_reservation_ledger`; כל פעולה נרשמת גם ביומן audit.
- קיימת אפשרות ליצור נתוני דוגמה רק במסד ריק, ורק לחשבון בעלים.

לפני חיבור נתוני עבודה אמיתיים יש לבצע בדיקת עומס מול המסד החדש בלבד. אין לייבא או לסנכרן נתוני ייצור מהמערכת הראשית בלי אישור מפורש וגיבוי מאומת.
