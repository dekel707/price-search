# הזמנות איתן — פיילוט מבודד

זהו פורטל נפרד עבור איתן. הוא קורא מחירון, לקוחות, שריונים וגיול מהמערכת הראשית דרך גשר שרתי קריאה־בלבד, ושומר את טיוטות ההזמנות שלו במסד PostgreSQL עצמאי עד לאישור הבעלים.

## גבולות ההפרדה

- אין שימוש ישיר ב־`DATABASE_URL`, `BLOB_READ_WRITE_TOKEN`, `/api/state` או ב־Vercel Blob של המערכת הראשית מצד הדפדפן או מסד הפורטל.
- כל טבלת נתונים בפורטל מתחילה ב־`partner_` ונמצאת במסד PostgreSQL חדש שהוגדר רק עבור הפרויקט הזה.
- נתוני המפרט מגיעים ב־GET מ־`TEAM_PORTAL_CATALOG_URL`. נתוני המחירון, הלקוחות, השריונים והגיול מגיעים רק מגשר `TEAM_PORTAL_MAIN_SYNC_URL` עם סוד שרתי (`TEAM_PORTAL_MAIN_SYNC_SECRET`).
- הפורטל לעולם אינו כותב ישירות למערכת הראשית. רק דקל יכול לאשר בתור "הזמנות איתן" באתר הראשי; באישור נוצרת הזמנה אחת עם גיבוי, והיא מסווגת לפי שעת האישור.
- האתר הראשי מציג את התור ומאשר דרך `TEAM_PORTAL_OWNER_BRIDGE_SECRET`. כל הסודות נשארים בשרתים בלבד; אין ללקוח או לדפדפן גישה אליהם.

## הגדרה לפריסה

1. יוצרים ב־Vercel **פרויקט חדש** עם Root Directory `partner-portal`.
2. יוצרים מסד PostgreSQL חדש לפרויקט החדש בלבד (Neon או Supabase) ולא מחברים שום אינטגרציית Storage קיימת.
3. מגדירים את כל המשתנים שב־`.env.example` רק בפרויקט החדש.
4. מגדירים את ה־Cron היומי ב־`vercel.json`; Vercel משתמש ב־`CRON_SECRET` כדי לחתום על הבקשה, והבקשה יוצרת צילום מצב במסד החדש.
5. הבעלים נכנס עם `TEAM_PORTAL_OWNER_PIN`; איתן נכנס עם `TEAM_PORTAL_EITAN_PIN`.

## הגנות נתונים

- לפני ואחרי כל פעולה משנה נתונים נוצר צילום מצב מלא בטבלת `partner_backups`.
- בזמן הכנת ההזמנה הפורטל מסמן את השריון בלבד; בעת האישור במערכת הראשית נבדקת יתרה עדכנית וההפחתה מתבצעת במסגרת שמירה אטומית ומגובה.
- לפעולה יש סטטוס `processing` קצר, כך שלחיצה כפולה או ניסיון חוזר אינם יכולים ליצור הזמנה ראשית כפולה. כל פעולה נרשמת גם ביומן audit.
- קיימת אפשרות ליצור נתוני דוגמה רק במסד ריק, ורק לחשבון בעלים.

לפני חיבור נתוני עבודה אמיתיים יש לבצע בדיקת עומס ללא אישור הזמנות. אין לייבא מסד נתונים מהמערכת הראשית; הקריאה החיה היא דרך הגשר המצומצם בלבד.
